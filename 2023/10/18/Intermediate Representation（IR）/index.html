Intermediate Representation（IR）

1. Compilers and Static Analyzers

编译器将源代码（Source code） 转换为机器代码（Machine Code）（在转换过程中也要报错）。其中的流程框架是：

  • 词法分析器（Scanner），结合（Regular Expression）正则表达式，通过词法分析（Lexical Analysis）将 source code 生成 Tokens，继续分析；没有通过词法分析要报错；

  • 语法分析器（Parser），结合上下文无关文法（Context-Free Grammar），通过语法分析（Syntax Analysis），将 token 解析为抽象语法树（Abstract Syntax Tree, AST）；

  • 语义分析器（Type Checker），结合属性文法（Attribute Grammar），通过语义分析（Semantic Analysis），将 AST 解析为 Decorated AST；

  • Translator，将 Decorated AST 翻译为生成三地址码这样的中间表示形式（Intermediate Representation, IR），并基于 IR 做静态分析（例如代码优化这样的工作）；Translator转化器生成IR，静态分析器来分析IR；

    经过以上步骤（前端），静态分析器会分析IR，为什么需要以上步骤，为什么不直接拿 source code 做静态分析？

    这是因为我们得先确保这是一份合格的代码，然后再进行分析。分析代码合不合格，这是 trivial 的事情，由前面的各种分析器去做就行了，静态分析器分析的是 non-trival properties（与程序运行时行为相关的性质）。

  • Code Generator，将 IR 转换为机器代码。

2. AST vs IR

AST和IR的不同，如下图，其中IR的compact and uniform指没有了AST的冗余信息，更简洁；IR很重要的特点是包含控制流信息。

为什么在静态分析的时候，使用 IR 而非 AST 呢？

这是因为：

  • AST 是 high-level 且接近语法结构的，而 IR 是 low-level 且接近机器代码的。
  • AST 是依赖于语言的，IR 通常是独立于语言的：三地址码会被分析器重点关注，因为可以将各种前端语言统一翻译成同一种 IR（具有语言无关的性质）（翻译成的一种统一的中间表示格式） 再加以优化。
  • AST 适合快速类型检查，IR 的结构更加紧凑和统一：在 AST 中包含了很多非终结符所占用的结点（body, assign 等），而 IR 中不会需要到这些信息。
  • AST 缺少控制流信息（看不出循环信息），IR 包含了控制流信息：AST 中只是有结点表明了这是一个 do-while 结构，但是无法看出控制流信息；而 IR 中的 goto 等信息可以轻易看出控制流。
  • 因此 IR 更适合作为静态分析的基础。

3. IR: Three-Address Code(3AC)

三地址码（3-Address Code）通常没有统一的格式。在每个指令右侧最多只有1个operator操作符（一般引入临时变量）。

IR：一种中间表示格式，一种形式！！！

三地址码为什么叫做三地址码呢？因为每条 3AC 至多有三个地址。而一个「地址」可以是（address有三个形式）：

  • 变量名 Name: a, b
  • 常量值 Constant: 3
  • 编译器生成的临时变量 Compiler-generated Temporary: t1, c

常见的 3AC 包括：

  • x = y bop z：双目运算并赋值，bop = binary operator 二元操作符
  • x = uop z：单目运算并赋值，uop = unary operator 一元（负、取反、类型转换）
  • x = y：直接赋值
  • goto L：无条件跳转，L = label
  • if x goto L：条件跳转
  • if x rop y goto L：包含了关系运算的条件跳转，rop = relational operator（>, <, ==, >=, <=, etc.)

4. 3AC in Real Static Analyzer: Soot

以上的 3AC 比较抽象，来看看现实中的 3AC。

Soot 是 Java 流行的静态分析器。

Soot 的 IR 是 Jimple: typed 3-address code。

Jvm 中的四种主要方法调用：

  • invokespecial: call constructor(调用构造函数), call superclass methods(调用父类方法), call private methods (私有方法)
  • invokevirtual: instance methods call (virtual dispatch) (派生)
  • invokerinterface: cannot optimization(不能优化<与invokevirtual相比>), checking implementation
  • invokestatic: call static methods (调用静态方法)
  • 另外：Java 7: invokedynamic –> Java static typing, dynamic language runs on JVM (动态解析出需要调用的方法，然后执行)

尖括号中：==> Method signature: class name: return type method name(parameter1 type, parameter2 type, ……)

默认构造函数函数没有名字 统一叫< init > 。

5. Static Single Assignment(SSA)

SSA：经典IR转换格式。

SSA比较古老，就是给每个定义一个新的名字。

所谓静态单赋值（SSA），就是让每次对变量x赋值都重新使用一个新的变量xi，并在后续使用中选择最新的变量。

3AC        |  SSA

p = a + b  |  p1 = a + b

q = p - c  |  q1 = p1 - c

p = q * d  |  p2 = q1 * d

q = p + q  |  q2 = p2 + q1

但是这样一来，肯定会因为不同控制流汇入到一个块，导致多个变量备选的问题：

这里解决的办法就是使用一个合并操作符$\phi$（phi-function），根据控制流的信息确定使用哪个变量。

tips（两个特征）：

每个variable都有自己的一个定义；

多重即引入（phi-function）；

为什么要用 SSA 呢？

  • 控制流信息间接地集成到了独特变量名中
    • 如果有些对控制流不敏感的简化分析，就可以借助于 SSA
  • 定义与使用是显式的
    • 更有效率的数据存取与传播，有些优化在基于 SSA 时效果更好（例如条件常量传播，全局变量编号等）

为什么不用 SSA 呢？

  • SSA 会引入过多的变量和 phi 函数
  • 在转换成机器代码时会引入低效率的问题

6. Basic Blocks(BB)

将3地址码转为CFG后进行静态分析。

BB是CFG的结点。

CFG 的一个结点可以是一条单独的 3AC，但是更常见的是一个基本块（Basic Block）。

BB是满足下列性质的 最多 的连续3地址指令：

只有一个入口，这个入口是BB的第一条指令；
只有一个出口，这个出口是BB的最后一条指令；

建立BB的方法：

输入：程序 P 的一系列 3AC

输出：程序 P 的基本块

（1）决定leader（入口）：

  • 第一条指令是一个leader ;
  • 是条件/无条件跳转指令的target指令的，是leader;
  • 在条件/无条件跳转指令之后的指令，是leader;

（2）构建BB：

  • 一个基本块就是一个 leader 及其后续直到下一个 leader 前的所有指令。

7. Control Flow Graphs(CFG)

BB是结点，除了基本块，CFG 中还会有块到块的边，怎么添加边？

  • blockA和blockB之间有边当且仅当：

    • A的出口是条件/无条件跳转指令，这个跳转指令是B的入口；

    • 在原指令顺序中B紧跟着A，并且A的出口不是无条件跳转指令；

  • 一般地，将BB中的跳转指令的label要替换成BB，

    eg: goto (i) 替换成 goto B

  • 如果A、B之间有边，则称A是B的predecessor前驱，B是A的successor后继。（前驱与后继都可以有多个）

  • 除了构建好的基本块，一般会额外加2个结点，「入口（Entry）」和「出口（Exit）」：

    • 这两个结点不对应任何 IR（并不是程序可执行的IR，引入只是为了设计算法时方便后面静态分析算法的设计初始化等等）；
    • 入口有一条边指向 IR 中的第一条指令（An edge from Entry to the BB containing the first instruction of IR）；
    • 如果一个基本块的最后一条指令会让程序离开这段 IR，那么这个基本块就会有一条边指向出口（An edge to Exit from any BB containing an instruction that could be the last instruction of IR）；

控制流分析（Control Flow Analysis）通常指的是构建控制流图（Control Flow Graph, CFG），并以 CFG 作为基础结构进行静态分析的过程。
