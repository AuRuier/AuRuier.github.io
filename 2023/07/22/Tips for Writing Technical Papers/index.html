Tips for Writing Technical Papers

链接（论文写作）：《Tips for Writing Technical Papers》

（前提）已有计划的例子（虚构的）：

假设已设计了一种新的外部多路合并排序算法并进行了实验。算法把复杂度从 O(nlogn) 降到了 O(n)，前提是在结果中出现一些有界的 “未排序 “是可以接受的。目前计划将结果写成文章，提交给一个重要会议。

论文标题

  • 标题可长可短——eg：具有近似保证的线性时间外部多通道排序
  • 或短小精悍——eg：近似外部排序
  • 这里是一个一般长度的名字（外加一个可爱的名词）——Floosh：近似外部排序的线性时间算法

引言

引言至关重要！！！

当评审人读完引言时，他们可能已经初步决定是接受还是拒绝这篇论文。
如果引言吸引了普通读者，他们会继续阅读，反之则会将论文搁置一旁。

以下是斯坦福信息实验室专利的五点式引言结构：

  1. 问题是什么？
  2. 为什么它有趣而重要？
  3. 为什么这个问题很难解决？
  4. 为什么以前没有解决？(或者，以前提出的解决方案有什么问题？ 我的解决方案有什么不同？）
  5. 我的方法和结果的关键部分是什么？还包括任何具体限制。

然后写最后一段或一个小节：”贡献摘要”。以项目符号的形式列出主要贡献，并提及在哪些章节中可以找到这些贡献。

相关工作

长期存在的问题：相关工作应该放在论文开头还是结尾？

如果相关工作足够简短但又足够详细，或者如果必须立即对以前的工作采取强硬的辩护立场，那么就应该在开头介绍。在这种情况下，”相关工作 “可以是引言末尾的一个小节，也可以是单独的第 2 节。

如果相关工作可以在早期（导言或前言中）快速总结，或者需要对论文的技术内容进行充分的比较，那么相关工作可以作为导言结尾的一个小节，或者作为单独的第 2 节。在这种情况下，”相关工作 “应放在 “结论 “之前，也可以放在 “讨论和相关工作 “这一节中。

正文

指导原则

  1. 在读者读完第 3 页（即论文读完四分之一时）时，应已阐明了一项明确的新的重要技术贡献。
  2. 论文的每一部分都应讲述一个故事。(但是，不要掉进一个常见的陷阱，即讲述你如何得出结果的整个故事。只需讲述结果本身的故事即可）。故事应该是线性的，让读者每一步都参与其中并期待下一步。（故事中不应出现重大插曲–这些插曲可放在附录）

除了这些适用于每篇论文的指导原则外，正文的结构也因内容而异。重要的组成部分包括：

  • 例证：在可能的情况下，在整篇论文中使用一个运行示例。可以作为引言末尾的一个小节，也可以单独作为第 2 或第 3 节（取决于相关工作）。
  • 序言：本节位于引言之后，也可能是相关工作和/或运行示例之后，用于设置不属于技术贡献的符号和术语。这一部分的一个重要作用是界定并非原创但论文需要的材料。（要简明扼要–记住准则 1）
  • 内容：论文的主要内容包括算法、系统描述、新语言结构、分析等。尽可能使用 “自上而下 “的描述方式：读者应该能够看到材料的走向，而且他们应该能够跳过前面的内容，但仍然能够理解其中的意思。

性能实验

性能实验应该测量什么？（这里提到了一些可能）

  • 纯运行时间
  • 对重要参数的敏感性
  • 各方面的可扩展性：数据大小、问题复杂性……

性能实验应该显示什么？（这里提到了一些可能）

  • 绝对性能（即可接受/可用性）
  • 与原始方法的相对性能
  • 与以往方法的相对性能
  • 不同建议方法之间的相对性能

结论

一般来说，一个简短的总结段落就可以了，在任何情况下，该段都不能简单地重复摘要或引言中的材料。在某些情况下，可以通过引用定量性能结果等方式，使原来的主张更加具体。

未来工作

这部分非常重要！！论文的部分价值在于展示其工作如何设定了新的研究方向。

eg：如果正在积极开展后续工作，请注明。例如”我们目前正在将算法扩展到……等等等等。

反之，也要注意有些研究人员会在 “未来工作”部分寻找研究课题。

致谢

感谢以任何方式做出贡献的人：通过讨论、对草稿的反馈、实施等。如果对是否将某人包括在内有疑问，那就将其包括在内。

引用

努力使所有引文完整一致。不要随意从网上复制不一致的 BibTex（或其他）条目就了事。仔细检查您的最终书目，确保每个条目看起来都是正确的。

附录

附录只应包含详细的证明和算法。

经验法则：

  • 附录不应包含任何理解论文贡献所必需的材料。
  • 附录应包含大多数读者不会感兴趣的所有材料。
  • 语法和小范围表述问题。
  • 一般来说，建议撰写论文的人阅读 Strunk 和 White 所著的《The Elements of Style》。
  • 论文中的所有 “变量”（术语和符号）都应在使用前定义，而且只能定义一次(有时在长时间中断后，提醒读者某个定义是有用的）。全局性定义应归入 “前言 “部分；其他定义应在首次使用前给出。
  • 不要使用 “等”，除非其余项目完全显而易见。
  • 可以接受：应将阶段编号为 1、3、5、7 等。
  • 不可接受：衡量性能因素，如波动性、可扩展性等。
  • 切忌说 “出于各种原因”(eg：由于各种原因，我们决定不考虑该备选方案）。需要告诉读者原因！
  • 避免使用 “this”、”that”、”these”、”it “等非指代性词语。要求明确指出 “这 “指的是什么，以确保写作的清晰性。
  • 斜体用于定义或引语，而不是用于强调（格里斯的忌讳）。在写作时，上下文本身就足以起到强调作用。
  • “that “是限定词，”which “是非限定词。
    eg：

    The algorithms that are easy to implement all run in linear time.

    The algorithms, which are easy to implement, all run in linear time.

机械

  • 请务必对您的最终论文进行拼写检查。
  • 对于草稿和技术报告，请使用 11 点字体、宽间距、1 英寸页边距和单栏格式。
  • 在草稿和最终成稿中，图表字体应与论文正文所用字体大小大致相同。
  • 表格、数字、图表和算法应始终放在一页或一栏的顶端，而不是正文中，除非它们非常小，与论文的流程相吻合。
  • 每个表格、数字、图表或算法都应与其第一个参考文献位于同一页，或位于下一页。
  • 在最终提交或发表论文之前，请打印出来看一看。

版本和分发

许多论文都有一个已提交（随后发表）的会议版本，以及一个 “完整论文 “的网络技术报告。

在可能的情况下，论文全文只包括会议版和附录。除了会议论文集之外，论文全文应该是唯一公开的版本，它应该与最新（最终）的会议版本相协调，对论文全文的修改应该始终覆盖所有可公开访问的以前版本。

论文一完成就应该放到网上。论文应注明日期，并可作为技术报告进行引用（千万不要在论文刚刚提交的时候就在上面贴上会议版权声明）。
